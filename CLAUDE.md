# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A DSPy pipeline that turns a one-line idea into a complete, review-ready video **story package**: idea → theme → story structure → raised stakes → four script formats, plus a title/thumbnail concept and a YouTube description. The defining property: every stage is **generated by one model and graded by an independent model on a different provider**, so the judge can never inflate its own work.

## Commands

Python 3.11. On Windows the interpreter is `.venv/Scripts/python` (a fresh clone has **no** `.venv` yet — create it before any live run or test).

```bash
python -m venv .venv
.venv/Scripts/python -m pip install -r requirements.txt
# cp .env.example .env  -> fill ZAI_API_KEY and OPENROUTER_API_KEY (only needed for LIVE runs)
```

Running the pipeline (`run.py` is the single entry point; every run is scoped to a `--channel`):

```bash
.venv/Scripts/python run.py new-channel demo-sports                      # scaffold channels/demo-sports/
.venv/Scripts/python run.py chain --channel demo-sports                  # whole pipeline over the seed queue
.venv/Scripts/python run.py chain --channel demo-sports --brief "..."    # one brief, end to end
.venv/Scripts/python run.py chain --channel demo-sports --brief "..." --dry-run   # offline wiring test, no keys
.venv/Scripts/python run.py generate --channel demo-sports --stage idea --brief "..."   # one stage, live
.venv/Scripts/python run.py manual   --channel demo-sports --stage idea --file manual_input.json  # hand-authored content+scores, no keys
.venv/Scripts/python run.py review   --channel demo-sports              # what's awaiting review (or --all across channels)
.venv/Scripts/python run.py learn    --channel demo-sports --stage idea # retrain that stage's generator from accepted examples
.venv/Scripts/python run.py package  --channel demo-sports              # assemble a video's deliverables into one doc
.venv/Scripts/python run.py cleanup  --channel demo-sports              # archive superseded draft revisions
```

Tests (all offline — no test ever makes a network/LLM call):

```bash
.venv/Scripts/python -m pytest -q                                  # full suite
.venv/Scripts/python -m pytest tests/test_pipeline_defects.py -q   # one file
.venv/Scripts/python -m pytest -q -k append_lock                   # one test by name
```

`pytest.ini` routes `--basetemp` to `c:/tmp/pytest-dspy` because the default temp root is not writable in this environment. There is no configured linter/formatter.

## The three run modes

Every path through the orchestrator is one of these — keep them straight:

- **live** — real paid API calls. Requires both keys. Only here does the engine hard-fail on a missing stage standard or a named-but-missing channel profile (a placeholder would mean money spent on nothing).
- **dry-run** (`--dry-run`) — two offline `DummyLM`s; exercises all the wiring (handoff, naming, auto-promote, reject path) with placeholder content. No keys.
- **manual** / scripted — your own content + your own gate scores, fed through the *real* engine via `DummyLM`. No keys. Use to test the loop on hand-authored material.

## Architecture

### The #1 rule: evaluator independence

The evaluator/reevaluator **must run on a different LLM provider and model** than the generator/iterator. `orchestrator.assert_separation` checks this at startup and **refuses to run** if they share an LM instance or model string. This is the invariant that keeps scores honest — do not weaken it. Current wiring in `config.py`:

- Generator / iterator → Z.ai GLM (`openai/glm-5.2`, `ZAI_API_KEY`)
- Evaluator / reevaluator → OpenRouter (`openrouter/google/gemma-3-27b-it`, `OPENROUTER_API_KEY`)

The same separation holds even while the system *learns*: `pipeline/metric.py` scores optimization with the separate evaluator too.

### A generic engine over "stages"

This is a multi-stage **engine**, not a single generator. A *stage* (`idea`, `theme`, `story`, `stakebake`, the four `script*` formats, `packaging`, `description`) is fully defined by three things, and the engine never changes when you add one:

1. Two standards: `rules/standards/<stage>_generator.md` + `<stage>_evaluator.md`
2. Three DSPy signatures in `pipeline/signatures.py` (generator / gate / iterator)
3. One `Stage` registry entry in `pipeline/stages.py`

The engine — these files — is generic and is what you **do not** touch when adding a stage: `orchestrator.py`, `naming.py`, `logging_setup.py`, `render.py`, `router.py`, `metric.py`, `generator.py`, `evaluator.py`, `iterator.py`, `reevaluator.py`, and the thresholds/paths in `config.py`. See `ADDING_A_STAGE.md` for the full recipe.

Consistency rules that aren't obvious — **MANDATORY**, and a change is not "done" until they are verified:
- A stage's `weights` dict in `stages.py` **must sum to 100** (`SCORE_SCALE`); any check scoring 0 forces a REJECT.
- The written standard (`rules/standards/*.md`) and the model-enforced version (signature docstrings + `OutputField(desc=...)` in `signatures.py`) must say the same thing — **two copies of one contract. If you change one, you change the other in the SAME pass — never one without the other.** (This is the single most common way this repo silently rots: editing a standard and forgetting its signature, or vice versa.)
- For every stage: `#weights == #labels == #gate score-fields`, and each gate `score_k`'s stated max **equals** its stage weight.

**Verification gate — run BEFORE calling ANY standard/signature/stage change "done":** `tests/test_signature_weight_sync.py` enforces all three rules above automatically across every stage, so `.venv/Scripts/python -m pytest -q` must be **green** before a change is finished. It fails the instant a signature drifts from its weights. Never claim a change is done without a green suite — and a `.md` edit with no matching signature edit (or the reverse) is, by definition, not done.

### The orchestrator loop (`pipeline/orchestrator.run`)

Generic over a `Stage`:

1. **Gate** — generate a candidate, run the gate. REJECT → save to `rejected/`, regenerate a brand-new one up to `MAX_GEN_ATTEMPTS` (5). PASS → assign/inherit a story id and proceed.
2. **Score + iterate** — improve the passing artifact up to `MAX_ITER` (3) times toward `TARGET_SCORE` (95). The critique tells the iterator to raise *only* the checks below full marks.
3. **Finalize** — a final independent re-evaluation. Still PASS and ≥ target → `ready_for_review`; otherwise → `parked`. **Nothing generated is ever deleted** (Project Rule 3); status lives in the filename `<slug>_<NUMBER>_v<VERSION>_<STATUS>.json`.

Some stages are hybrids: `verdict_floor` (PASS at/above a floor instead of all-checks-pass), `penalty_points` (e.g. a jargon penalty), `gate_reads_package` (the gate is handed the assembled upstream package to verify the script actually delivers it), and `deep_channel` (the *full* channel profile feeds the stage, not just the positioning spine).

### The chain (`pipeline/chain.py`)

`chain.run_chain` drives one brief through everything:

```
idea → theme → story → stakebake          (the spine: auto-promote each on target, STOP at the first weak link)
  → script, script_long, script_screenplay, script_podcast   (4 independent formats off the accepted stakebake)
  → [promote script_long]                 (only if it reached ready_for_review)
  → packaging, description                 (both built off the PROMOTED long-form script — it holds the whole story)
```

Auto-promote = a copy into `accepted/` so the next stage can pull it, **minus** the trainset write (auto-promotes are not human-accepted, so they must not retrain the generator). If `script_long` doesn't reach `ready_for_review` it can't be promoted, so packaging/description are *skipped* (reported honestly), not run — the chain never lets the orchestrator hit its "no accepted upstream" error.

Every artifact of one video shares a single `story_id` (`ST-####`, minted by the first stage and threaded forward); the formats stay separable because each is its own `stage`.

### Per-channel workspaces

A channel is a self-contained production line under `channels/<name>/`. `config.paths_for(channel)` resolves the whole workspace (path-traversal guarded; no `--channel` → a shared `_sandbox`). The channel's `profile.md` carries the audience + voice and rides into every generator. A `<!-- SPINE:START -->…<!-- SPINE:END -->` block is the tight positioning fed to *every* stage; deep stages get the whole profile. Layered into each generator prompt: channel profile + `craft/principles.md` (genre-agnostic doctrine) + the stage standard.

Runtime data lives under each channel and is gitignored — **except** `channels/*/review/human_review.md`, the append-only review journal, which is the durable source of truth and *is* tracked. Artifacts land in `channels/<name>/outputs/{drafts,accepted,rejected,parked,archived}/` as paired `.json` (pipeline) + `.txt` (human) files.

### Review & learning

Human review now happens **in conversation** (via the `review-logging` skill), which writes verdicts + reasoning to `review/human_review.md` and promotes/rejects directly. `pipeline/router.py` is a legacy batch path kept for back-compat. Accepted artifacts append to `memory/trainset.jsonl`; `pipeline/learn.py` compiles a better stage generator from those real decisions via a DSPy optimizer, graded by the separate evaluator.

`rules/PROJECT_RULES.md` is the standing doctrine (closed system — no code reads or writes outside this folder; evaluator independence; versioned-never-overwritten; the status state machine; the learning loop). Read it before changing engine behavior.
